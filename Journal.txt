		"Impulse Strength". The former seems to do nothing, but I'm not going to take the time to chase it down.

		> Ok I lied ... I took a couple minutes to scan through the UE code and discover the messiness that makes this so confusing.
			1. Take a peek at how WE have set up trhe barrel. Our callback is calling the RadialForceComponent's "FireImpulse()" method directly
				- we never set anything up that will apply the "ForceStrength" force within the components Tick !
			2. So, what the heck is going on here?
				- IT'S CRAP !
				- UE has created the URadialForceComponent with too many responsibilities within a single class !
				- There's actually 2 completely separate interfaces here.
					> "Impulse" Approach : You can set values for the impulse approach and "wire it up" to the "FireImpulse" method. You can
						set the flag to apply the same delta-V to all objects (regardless of their mass). This is NOT physical, but people us
						it alot because it's easier to tune than understanding the physics.
					> "Force" Approach : You can set the ForceStrength , then Activate (or auto-Activate) this component and forces
						will be applied through the logic in the TickComponent. This is FANCY, and includes ability to Tick at specified
						intervals (pulse). Note that for this case the MASS of the objects are taken into consideration since it applies forces
						instead of impulses flagged to apply the same delta-V for all objects.

					> NOTE: You can set up BOTH approaches to work within a single instance ! FANCY - but makes these interfaces quite confusing.

				- I created a BP example "RadialForceBlackHoleBP" : Use this as an example of how things can be "wired up" to implement
					both approaches simultaneously. This "Black Hole" continuously "pulls objects" into it, but every 5 seconds there is
					an impulse applied to all the "trapped" objects that pushes them outward.

					> NOTE: The SCharacter only gets pulled into the black hole whe he jumps (probably physics is not simulating when he's
						in contact with the ground so the controller has complete control of his movement)

Lecture 7.5 : Dealing Damage with Attributes

	Course design is slightly different!

	Course Design (up to end of Lecture 7):
		Bumping into the Barrel causes it to explode.
		Hitting the Barrel with magic projectiles does NOT cause it to explode
		When you bump into the Barrel and it explodes, damage is only applied to the Character that hit the barrel
		Neighboring Actors may "feel" the impulse and move radially, but receive no damage

	My Design:

		Bumping into the Barrel does NOT cause it to explode... just the usual physics
		Hitting the Barrel with magic projectiles DOES cause it to explode
		When the Barrel explodes, damage is applied to all actors that are affected by the Radial Force Component
		So, damage is applied to a Character  standing close to the barrel if the turret cannon hits the barrel
		Damage is applied by a custom SRadialForceComponent placed onto the barrel 
			> derived from URadialForceComponent
			> overrides FireImpulse , collects unique SAttributeComponents in the overlap query and adjusts health
			> RadialForceDamage value is exposed as a property

		+ OK, now I see ... Here's the poop (from Claude):

			meta = (BindWidget) is a UMG (Unreal Motion Graphics) specifier that creates a binding between
			a C++ UPROPERTY and a widget that exists in a Blueprint-created UUserWidget.
			Concretely, it means:

			You declare the widget variable in C++ with BindWidget
			You create a widget of the same name and same type in the UMG designer (the Blueprint visual editor)
			Unreal automatically connects the two — your C++ variable becomes a valid pointer to that designer-created widget at runtime

			So if you have USizeBox* ParentSizeBox in C++, you must have a SizeBox widget named exactly ParentSizeBox in your UMG
			Blueprint layout. If the names don't match, you'll get a compile/load error (or warning, depending on configuration).
			The practical benefit is that you can then manipulate that designer-placed widget from C++ code — changing its properties,
			binding events to it, etc. — while still being able to visually lay it out in the UMG designer.
			It's essentially the UMG equivalent of how VisibleAnywhere exposes components to the Blueprint editor, except BindWidget goes
			the other direction: the widget lives in the Blueprint designer, and C++ gets a pointer to it.

		+ TRICKY !
			So, you can declare a pointer variable in your C++ without address how the object gets instantiated, then add an object
			to your widget of exact same TYPE and with exact same VARIABLE NAME, and it will magically bind to the one you've added.

						if(ApplyDamage(InstigatorActor, TargetActor, DamageAmount))

							UPrimitiveComponent* HitComp = HitResult.GetComponent();
							if(HitComp && HitComp->IsSimulatingPhysics(HitResult.BoneName))
